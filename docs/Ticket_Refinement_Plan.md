# 工单详情页重构与维修报告需求规范 (v1.0)
**状态**: 需求确认
**最后更新**: 2026-03-07
**关联路径**: `docs/Service PRD_P2.md`

## 1. 工单详情页 UI 重构 (Ticket Detail UI Refinement)

### 1.1 核心原则：动静分离
将“流程现状（动态球权）”与“资产事实（静态属性）”在视觉上彻底解耦。

### 1.2 顶部 Header：动态球权组件 (Ownership Component)
*   **组件名称**: `OwnershipBadge`
*   **位置**: 替换原有的静态 `[处理中]` 标签。
*   **显示逻辑**: 
    - `[ 节点名称 · 部门名 · 具体负责人 ]`
    - **示例**: `[ 物理维修 · OP · 张工 ]`
    - **待认领状态**: 若无负责人，显示 `[ 节点名称 · 部门名 · 🚩 待认领 ]`，提供一键认领功能。
*   **视觉规范**: 沿用 Kine Yellow / Kine Blue 语义色，配合磨砂质感 (Glassmorphism) 以突出当前进度。

### 1.3 优先级重新定位 (Priority Re-anchoring)
*   **动作**: 将 `P2` 标签从顶部 ID 旁移除。
*   **位置**: 迁入“基本信息”卡片的首行首位。
*   **视觉**: 使用实色高亮 Tag，体现 SLA 的契约属性。

### 1.4 基本信息区重排 (Basic Info Restructuring)
*   **移除字段**: `指派给` (该功能已上移至 Header 动态球权)。
*   **布局逻辑 (Grid)**:
    - **行 1 (SLA & Audit)**: `优先级 (Priority)`, `创建时间 (Created At)`
    - **行 2 (Asset Info)**: `产品型号 (Product)`, `序列号 (SN)`
    - **行 3 (Client Info)**: `客户 (Account (Link))`, `报修人 (Reporter)`
    - **行 4 (Internal)**: `提交者 (Submitted By)`

## 2. 维修报告自动生成机制 (Repair Report Draft Mechanism)

### 2.1 业务流程映射 (Logic Mapping)
系统应在 OP 提交维修结论后，自动拉取技术数据，为 MS 准备一份专业对客的报告草稿。

| 来源 (OP 填报内容) | 转化目标 (客户报告内容) | 说明 |
| :--- | :--- | :--- |
| **Troubleshooting (排故记录)** | **故障检测结论 & 维修执行摘要** | 系统提取文本，MS 进行商务润色。 |
| **Parts Table (配件清单)** | **费用明细 - 配件部分** | 自动拉取 SKU 名称及价格，形成清单。 |
| **消耗工时 (Labor Hours)** | **费用明细 - 人工服务费** | 根据工时级别与时长自动计算小计。 |
| **现场照片 (Attachments)** | **诊断证据 (Evidence)** | 嵌入报告作为收费的核心支撑证据。 |

### 2.2 报告草稿核心组成
1.  **标识区**: Kinefinity Logo, 工单 ID, 报告日期。
2.  **资产/客户信息**: 型号, SN, 客户, 保修判定 (手动/系统判定)。
3.  **技术说明**: 
    - **Diagnostic Findings**: 故障根本原因说明。
    - **Actions Performed**: 采取的硬件更换/固件升级措施。
4.  **财务清单 (Itemized Statement)**: 
    - 配件费 (零件名、数量、小计)。
    - 人工费 (服务等级、时长、小计)。
    - 总计金额 (Total Due)。
5.  **质保承诺**: 本次维修提供的保修期 (默认 90 天)。

### 2.3 交互逻辑
1.  **生成**: OP 提交给 MS 时，后端自动初始化 PDF/Web 预览。
2.  **编辑器**: MS 客服可以在详情页中打开该草稿，修改对客描述或手动调整金额 (折扣)。
3.  **分发**: MS 确认后，通过邮件或系统推送发送至客户预览链接。
