## 2. 维修报告自动生成机制 (Repair Report Draft Mechanism)

### 2.1 业务流程映射 (Logic Mapping)
系统应在 OP 提交维修结论后，自动拉取技术数据，为 MS 准备一份专业对客的报告草稿。

| 来源 (OP 填报内容) | 转化目标 (客户报告内容) | 说明 |
| :--- | :--- | :--- |
| **Troubleshooting (排故记录)** | **故障检测结论 & 维修执行摘要** | 系统提取文本，MS 进行商务润色。 |
| **Parts Table (配件清单)** | **费用明细 - 配件部分** | 自动拉取 SKU 名称及价格，形成清单。 |
| **消耗工时 (Labor Hours)** | **费用明细 - 人工服务费** | 根据工时级别与时长自动计算小计。 |
| **现场照片 (Attachments)** | **诊断证据 (Evidence)** | 嵌入报告作为收费的核心支撑证据。 |

### 2.2 报告草稿核心组成
1.  **标识区**: Kinefinity Logo, 工单 ID, 报告日期。
2.  **资产/客户信息**: 型号, SN, 客户, 保修判定 (手动/系统判定)。
3.  **技术说明**: 
    - **Diagnostic Findings**: 故障根本原因说明。
    - **Actions Performed**: 采取的硬件更换/固件升级措施。
4.  **财务清单 (Itemized Statement)**: 
    - 配件费 (零件名、数量、小计)。
    - 人工费 (服务等级、时长、小计)。
    - 总计金额 (Total Due)。
5.  **质保承诺**: 本次维修提供的保修期 (默认 90 天)。

### 2.3 交互逻辑
1.  **生成**: OP 提交给 MS 时，后端自动初始化 PDF/Web 预览。
2.  **编辑器**: MS 客服可以在详情页中打开该草稿，修改对客描述或手动调整金额 (折扣)。
3.  **分发**: MS 确认后，通过邮件或系统推送发送至客户预览链接。

## 3. 设计参考 (Visual Mockup)
*   **Mockup 路径**: `docs/ui_mockups/repair_report_mockup.png`
*   **设计原则**: 极其专业、信任感、深色模式、多语言支持。


OP部门（技术/运营）提供的是“硬核”的技术执行数据：Troubleshooting（排故过程）、Parts（消耗配件清单）、Labor（工时）。 但是，这份报告最终是要呈现给客户（即作为 MS 商务审核收费的依据，也就是向客户发出的 Repair Report / Service Quotation）。因此，它必须在“体现专业度”和“便于客户理解/付款”之间取得平衡。

针对你提到的这三部分核心输入，我认为这份生成的客户级维修报告草稿，应该包含以下结构和内容：

1. 基础信息区 (Header & Device Info)
这是合同和溯源的基石。

工单号 (Ticket No.): 比如 RMA-C-2603-002，方便客户对账。
日期 (Date): 报告生成日期。
客户信息 (Customer Info): 客户名称，联系人。
设备信息 (Equipment Info): 产品型号 (如 Eagle SDI)，序列号 (SN)。
2. 故障概述设定 (Issue Summary)
客户描述故障 (Reported Issue): 客户当初报修时说的症状（这能让客户知道我们在针对他的痛点解决问题）。
3. 技术诊断与执行 (Diagnosis & Action Taken) - 由 OP Troubleshooting 转化
这里需要把 OP 填写的内部排故过程，精简或转化为客户能看懂的**“技术说明”**。

检测结果 (Diagnostic Findings): 比如“经检测，确认 SDI 输出模块因静电击穿导致无信号，符合您描述的屏幕无显示症状。”
维修措施 (Actions Performed): 比如“拆解清理内部，更换 HDA 主板，重新刷入最新固件并校准色彩槽位。”
(系统建议)：内部界面上，OP 填写的 Troubleshooting 最好分两层：一层叫 内部技术备忘 (Internal Notes)，一层叫 对客说明 (Customer Facing Note)。系统抓取后者生成报告。
4. 费用明细清单 (Itemized Billing) - 核心收费依据
这是报告直接促成“收费”的核心部分，由 OP 填写的 Parts 和 工时 自动生成。

A. 备件费用 (Parts & Materials)
表格形式展示：配件料号 (SKU) | 配件名称 (如 HDA 主控板) | 数量 (Qty) | 单价 | 小计
B. 人工与服务费 (Labor & Services)
表格形式展示：服务项目 (如 高级硬件维修工时) | 时长/数量 | 工时费率 | 小计
C. 测试与物流 (其他可能费用)
比如回程运费 (Return Shipping)，如果有的话。
5. 财务汇总与保修判定 (Financial Summary & Warranty Status)
保修覆盖 (Warranty Coverage): 这是必须标明的。比如“此项故障被判定为人为/过保，需全额付费”或“配件费用免除，仅收人工费”。
总计金额 (Total Amount Due): 提供一个清晰的合计数字（根据客户区域决定是否体现税金等）。
6. 品控与质保背书 (QC & Guarantee)
给客户一颗定心丸，证明钱花得值：

测试结果 (QC Test Passed): 例如“设备已通过严格的出厂级老化测试及各项功能校验。”
维修保修条款 (Repair Warranty): “本次更换的配件享受 90 天质保。”
总结一下产生这个草稿的逻辑映射：

OP 的 Troubleshooting ➡️ 转化为 第3部分（检测结果与维修措施）
OP 填写的 Part 表格（比如HDA） ➡️ 转化为 第4部分A（备件费用明细）
OP 填写的 消耗工时 ➡️ 转化为 第4部分B（人工费明细）
从产品设计的角度，当 OP 点击【提交维修结果】时，系统拉取上述数据，套用一个 PDF 模板或网页预览视图，然后在流转节点变为 “MS 商务审核” 时，MS 的客服看到的首先就是这一份**“维修报价单/报告 草稿”**。MS 确认无误，甚至微调一些对外话术或给予折扣后，一键发送给客户请求付款。


第一步：OP（维修工程师）的后台输入操作
维修人员（比如 张工）修完机器后，在系统的页面里只填了最硬核的数据：

排故与维修记录 (Troubleshooting)：
"测了 SDI 没信号输出。拆机查了波形，发现是靠近外壳的视频输出隔离芯片被外部异常高压击穿了，连带 HDA 主板供电也不稳。直接换了 V3 版的 HDA 总成。刷了最新的 v7.1 固件，跑了 4 小时满载烤机，色彩拉满没问题，各项指标 OK。"

配件消耗表 (Parts Table)：
扫码添加：SKU: K2-HDA-V3 | 名称：HDA Mainboard Assembly | 数量：1 (系统自动拉取该 SKU 的备件价格：¥4,500)
工时消耗 (Labor)：
选项框选择：二级硬件维修 (L2 Hardware Repair)
时长填写：2 小时 (系统根据 L2 费率自动计算：2 * ¥400 = ¥800)
第二步：系统自动生成的【维修报告/报价单草稿】
当 OP 点击“提交”并把工单流转给 MS（商务审核）时，系统会基于上述输入，自动在这个 RMA 工单下生成一份类似如下格式的 PDF 预览版草稿。

MS 的客服打开工单，看到的将是这样一份结构化、对客友好的文档：

【设备与基本信息】

【Kinefinity 原厂质量保证 (Quality Guarantee)】 本次维修所更换的 HDA 主板总成 及其相关维修工艺，自设备发货之日起，享受 90 天有限质量保证。

这套设计的核心价值：
防止 OP 乱说话：OP 只需要关注“换了什么、修了多久”，系统将其框定在“内部记录”和“严谨的数值”里。
赋权 MS 做商务决策：MS 客服在审阅这份草稿时，可以：
润色 检测结果 的措辞（比如把 OP 写的比较口语的“烤机没问题”改成“已完成满载压力测试”）。
针对 VIP 客户（比如 Panavision），MS 可以直接在界面的草稿上点击“人工费”，将其修改为 ¥0 (100% 折扣 - VIP 客户关怀)。
客户极度清晰：客户收到这份报告（可以通过邮件链接或者导出的 PDF），能清楚地看到 “病因是什么 -> 换了什么零件 -> 零件多少钱 -> 工时怎么算的 -> 提供多久质保”，极大地降低了付款时的扯皮概率。
