## 一、背景

项目已进入试用准备阶段。产品开发有 PRD 文档作为团队同步基础，但面向最终用户的操作指南、就地提示和持久知识库仍需体系化建设。本文档规划知识库的内容体系、技术方案和工作流程。

## 二、三层文档金字塔

```
        ┌─────────────┐
        │  就地提示     │  ← 用户"正在做"的时候看到
        │ (Contextual)  │     Tooltip / Empty State / Badge
        ├───────────────┤
        │  操作指南      │  ← 用户"想学"的时候查阅
        │ (KB Articles)  │     知识库内的 SOP / 功能指南
        ├───────────────┤
        │  PRD / 设计文档 │  ← 开发团队内部参考
        │  (Dev Docs)    │     当前 docs/ 目录
        └───────────────┘
```

- **就地提示**：用户操作过程中的实时引导（Tooltip、空状态文案、首次发现提示）
- **操作指南**：知识库中的结构化文章，用户主动查阅
- **PRD/设计文档**：开发团队内部参考，不面向最终用户

## 四、就地提示 (In-App Contextual Help) 策略

### 4.1 空状态引导 (Empty State)
- 当"我的任务"为空时，显示"暂无待办，你可以去团队协作认领工单"
- 当"部门工单"为空时，提示"当前部门暂无工单"

### 4.2 首次功能发现 (First-time Hint)
- 用户第一次进入工单详情页时，短暂高亮进度条并提示"悬停查看当前球权"
- 用 `localStorage` 记录用户是否已看过

### 4.3 操作确认增强
- 删除工单前的确认弹窗附带"删除后仅管理员可从回收站恢复"
- 编辑核心字段时弹出的"修改原因"框，附带一行小字解释为什么要填

### 4.4 KB 文章内链
- 在工单创建弹窗的顶部或底部，链接到"如何创建 RMA 工单"的 KB 文章
- 在权限被拦截时，鼠标悬停显示"了解权限规则"链接

## 五、技术方案：本地 Markdown → 远程发布

### 5.1 文件结构
```
docs/kb/
├── rma-process-sop.md           # RMA 流程 SOP
├── ticket-permissions.md        # 工单权限规则
├── inquiry-quick-guide.md       # 咨询工单指南（待写）
└── ...
```

### 5.2 文件格式
每个 `.md` 文件使用 YAML frontmatter 声明元信息：

```markdown
---
title: "RMA 流程 SOP：球权在谁手里？"
slug: rma-process-sop
product_line: GENERIC
category: Manual
visibility: Internal
tags: ["RMA", "SOP", "Workflow"]
summary: "通过球权比喻，深入排摸 RMA 工单在市场、运营及财务部之间的流转逻辑。"
---

（正文 Markdown 内容...）
```

### 5.3 发布脚本
`scripts/publish_kb.sh`：读取 `.md` 文件，通过 SSH 将内容 INSERT/UPDATE 到远程 `knowledge_articles` 表。

**优势**：
- 本地可预览（VS Code Markdown 渲染）
- 版本控制（Git diff 追踪）
- 避免 SSH 命令行转义问题
- 支持批量发布

## 六、工作流程

### 6.1 新增文章流程
1. 我在 `docs/kb/` 目录下起草 `.md` 文件
2. Jihua 在 VS Code 中预览并审核内容
3. 运行 `./scripts/publish_kb.sh docs/kb/xxx.md` 发布到远程

### 6.2 修改文章流程
1. 修改 `docs/kb/` 下对应的 `.md` 文件
2. 运行发布脚本（脚本基于 slug 做 UPSERT）

### 6.3 批量发布
```bash
./scripts/publish_kb.sh docs/kb/*.md
```
